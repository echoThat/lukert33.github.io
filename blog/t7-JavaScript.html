Flexibility in JavaScript vs Ruby

Oct 31, 2014

A big question occured to me as I, a novice Rubyist, set out to learn JavaScript was this: where are all the Errors? This is quite the opposite of what I'd thought when first learning Ruby ("Why oh why are there so many Errors!?") But as I my use of Ruby has matured I've learned to appreciate its stern watch over my syntax.

Unlike Ruby, Javascript will permit my code to run through to completion unless the syntactical problems fall into a few eggregious categories. A call on a completely unestablished variable or function will put up a flag. What won't, however, is what Ruby calls (frequently, in my case) a TypeError. Let's say I found a string of numbers in a document, and want to put it into an array to analyze it. In either language, this will work: secret_codex = "12 67 8".split(" ") #-> ["12","67","8"]

So I've got a hunch about these numbers, and run them through a homebrewed cryptographic process of mine: (secret_codex[0] * secret_codex[1]) % secret_codex[2] The result, in JavaScript, is that they are passed through this as if each is a number, and not a string. It returns //->4

Ruby raises a descriptive Error:

TypeError: no implicit conversion of String into Integer

In Ruby, I have to convert each of the elements of secret_codex into a string before treating them like the numbers I want to treat them as.

Why is this not the case in JavaScript? In some of my initial coding in JS, I actually perceived the syntax to be more rigid. Ruby has many synonyms for the same methods, for instance, and it's vocabulary builds in more interchangeability with spaces and parentheses. Yet in running the actual code--the place where it counts, one might say--JavaScript is much more flexible.

Part of this, as Marijn Haverbeke writes in Eloquent JavaScript, is because early JS designers thought the flexbility would make it more accessible. There are many accounts of this not being the case. Many have found, like I did, that languages that are less permissable and call our Errors more frequently, are easier to learn.

I suspect the flexbility has a benefit for users, though. For so many applications that serve casual users, it's preferable to have something act weirdly than to have a larger portion of the program refuse to execute; the Internet is hardly free of wonky code, but it works. This, combined with JavaScript's more limited core vocabulary, makes it more crash-proof when displayed by the limitless number of browser configurations that will access a public website.

Still, in an effort to release code with no bugs for the end user to deal with, I know that as I use JavaScript I'll try to always be sure to keep its flexibility in check, even when I have to write an army of Rubyist Error messages myself.
