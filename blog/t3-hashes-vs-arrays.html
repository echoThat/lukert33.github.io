Arrays vs Hashes

October 3, 2014

How to Choose the Right Data Structure

What's an Array?

An array is the default data structure that you'll use in Ruby. It's a collection of objects, with each object automatically assigned a numbered index slot in the array. That it's a collection of objects isn't unique. Hashes are also collections, and in Ruby you can add any possible object to either a Hash or an Array.

What's unique about Arrays are those index slots. Every item in the array is assigned a number that identifies its index position, and the index remains attached to the item until you do something to change that. So in this array: new_array = ["cat", "carrot", 6] the string "carrot" is assigned to the second index. Because indexing in arrays starts at 0, you can call the carrot with new_array[1]. This illustrates the characteristic of Arrays that makes them the data structure of choice. Because of that automatic numerical indexing, Arrays have far more ways of being manipulated than Hashes. Because you know there is always a reliable index system, you can call on specific pieces and portions of the array, you can reassign items to indexes based on any criteria you imagine, and use that as a sorting technique. Essentially, an Array has all the power of mathematical principles on its side when it comes to determining how it is arranged.

Sexy...I know. So what do Hashes have to say for themselves?

What's a Hash?

A hash differs from an array in that there is no built-in index. Lacking that, Hashes cannot access the many functions that Arrays leverge due to their automatic numerical slots. If you need to call every other object in your data structure, or every eleventh, use an array. That would be new_array.each do |elevensies| next if index(elevensies) % 11 != 0 #do what you want to every 11th item end So, why ever use a Hash if you leave all that mathematical power behind? Well, you use it specifically when it's more useful to assign non-numerical indexes to the content of the array. In Hashes, the function of indexes is replaced by keys. When you build the Hash, you decide what each key is going to be. It's useful to think of Hashes as dictionaries. It's almost pointless to arrange a dictionary numerically--you don't say "Alaska" is the 90th entry in the dictionary, you say it's somewhere between "Aardvark" and "Apricot." Here's an example of a Hash construction:

new_hash = { :key => "value" :key_orange => "an orange value" :key_purple => "probably purple" :key_velvet => "a velvet value" }

So to return "a velvet value" in new_hash, you call new_hash[:key_velvet] As you can see, the format for calling the item by a key is the same as calling something by index in an array, e.g. new_array[1]

You use a Hash when it's more useful to apply a qualitative key that will always be affiliated with the object you assign it to (called the "value" assigned to the key). You give up the mathematical power of arrays, but the point of doing so is that the situation makes a descriptive key better. In a dictionary, you'd rather say that dictionary["Aardvark"] is "a nocturnal burrowing mammal with long ears, a tubular snout, and a long extensible tongue, feeding on ants and termites." Rather than assigning that definition to dictionary[24]

Hash versus Array

It comes down to whether you'd rather have a numerical index assigned to the objects in your data collection, or a more specific, qualatiative key. Because the numerical nature of indexes makes it easy to apply math to an Array, Arrays are the more common structure. Yet Hashes are indispensable in situations where math is simply beside the point.
