Ruby: Variable Scope

October 26, 2014

The Need-to-Know Basis of Ruby Objects

When you first learned Ruby, you learned how to declare a local variable. When you assign a value to x, as in x = 10, you are ensuring that only the code construct you are within--maybe a class definition, maybe a method, maybe just main in the program you are writing--knows about that variable

Local variables start with an underscore or lower-case letter, but they are not the only type of variable you can declare. They are the most discreet variables. That is, their scope is the smallest. A local variable's information will only be known to objects within the code construction (class, module, method, main, etc.) in which it was declared.

Because local variables are the most discreet, it is fitting that they are the first that new rubyists learn to use. In object oriented design, you always want to limit the amount of messages an object will respond to. This keeps the behavior of your objects predictable, and helps you diagnose bugs that arise from future changes.

The polar opposite of a local variable is a global variable. These always begin with a dollar sign. If you declare $x = 10, the entire program you are writing will have knowledge of what $x is. Global variables are used sparingly in object oriented design, as there is rarely a reason to create information that every object needs to have behavior for.

Constants are variables with a key feature: they are not meant to be changed. In ruby, you assign constants by starting the variable name with caps, e.g. CONSTANT = "green". By convention, constants are generally written in ALL CAPS. The main feature of constants in Ruby is that the Ruby interpreter will raise an objection if a program tries to change an already declared constant. You use them to contain information that should be accessed on a read-only basis. In terms of scope, constants act similarly to local variables, in that they are generally accessible within the code construct in which they were declared. That said, when a constant is declared outside of a class of module, it will have global scope.

Instance variables and class variables behave in similar ways. This is reflected in their notation. Instance variables, which are unique on a read-write basis to any given instance of a class, are declared with a single @ at the beginning, e.g. @height = 6. Using instance variables when you are creating or modifying a class allows each instance of that class to have a certain amount of information about itself, and all of that will be pre-categorized. Instance variables ca vary between instance of the same class of object.

Class variables, on the other hand, will always have the same value for any instance of a class. They are declared with two @ signs instead of one, e.g. @@language = Japanese. You can usae class variables to provide information that you know all instances of your class will need to have, but that you don't want other objects to have access to.

Generally, the best practice is to keep your variables limited, and only expand toward global scope when you must. Local variables and instance variables are, therefore, the primary variable type you'd turn to.
